### Stray seam at the content_stack boundary (real-screen-only artifact)

User reported (with screenshot) two darker 1px lines on the Options General page — one ~10px from the window's right edge, one ~10px from the bottom edge, meeting at a 90° angle near the bottom-right corner. 10px = the 2px `_OptionsPanel` margin + the 8px `_CONTENT_RADIUS` clearance added in the de-masking fix above — i.e. exactly `content_stack`'s boundary inside `content_col`. Pixel-scanned that exact boundary via `grab()` in the offscreen test harness and found perfectly uniform colour with no anomaly anywhere — confirming this is a real-compositor-only artifact `grab()` can't surface, the same class of testing blind spot as the earlier `setMask()` discovery, just on the rendering side instead of the masking side. At that boundary there were four independently-sized layers all painting the identical `SYS_BG_SECONDARY`: `content_col`'s `_Surface` fill, `_content_stack`'s own QSS background, the active page's `QScrollArea` QSS background, and that page's inner `_Surface`. Removed the two redundant middle layers — `_content_stack`'s `setStyleSheet(...)` call and `_section_scroll()`'s `QScrollArea` background declaration (kept `border: none` only) — since `content_col` and each page's inner `_Surface` already cover that area; under real DPI-scaled compositing, independently-rounded sub-pixel boundaries between QSS-styled and directly-painted layers of the *same* colour can leave a 1-device-pixel seam that's invisible in software-rendered `grab()` testing. All 8 option pages reverified clean post-fix in the offscreen harness (this only confirms no regression — it was never able to reproduce the artifact itself, so the real test is on the user's actual screen).

**User reported the seam was still present on a real screen after that fix.** Built a temporary `test_window.py` (burger menu → "Test Window", `TaskBar.test_window_requested`) to bisect: an exact copy of `OptionsWindow`'s shell, built up one layer at a time by importing the *real* functions from `options_view.py` rather than reimplementing them, so each step was a true apples-to-apples copy. Confirmed clean at every structural layer in sequence: (1) bare nav + content_col skeleton, (2) + the real `content_header`/`QStackedWidget` container and margins with a blank page, (3) + the real `_section_scroll()` (`QScrollArea` + inner `_Surface`) with zero cards, (4) + the actual General-page card content (`_ToggleSwitch`/`_setting_row`/`_card_wrap`, imported directly). All four were faultless on the user's real screen — ruling out every structural layer in `options_view.py`, including the two "redundant layer" removals above.

**Root cause, found by a much cheaper test the user suggested in parallel**: resizing `OptionsWindow` by +2% (820×640 → 836×653) made the horizontal line vanish; a further +2% on width only (→ 853×653) made the vertical line vanish too. This is a pure fixed-size/DPI-rounding coincidence — at the original 820×640, some pair of independently-laid-out widgets happened to round to different device pixels under the OS compositor; at other sizes they don't. Nothing structural was ever wrong. Rounded the confirmed-clean 853×653 to a tidier **850×650** and re-confirmed clean on a real screen at that exact value before keeping it (`options_window.py`'s `self.resize(850, 650)`, comment explains why the exact pixel dimensions matter and to re-check if this window's size ever changes).

The earlier redundant-layer removal (`_content_stack`/`QScrollArea` backgrounds) and the de-masking work are both still correct and worth keeping as cleanup/hygiene, but neither was the actual fix for this seam — the size change was. Corrected the `pyqt6-frameless-window` skill's mistakes-table entry accordingly: check for a fixed-size/DPI-rounding coincidence by nudging the window's `resize()` call first (cheap), before bisecting structure with a stripped-down test window (expensive). `test_window.py` and its menu entry are left in place for now (still marked `TEMPORARY` in code) at the user's preference, in case further experiments are wanted.

**Follow-up cosmetic tweaks to `OptionsWindow`**: halved `_OptionsPanel`'s border stroke (1.5px → `_BORDER_WIDTH = 0.75`, with `_BORDER_INSET`/`_BORDER_RADIUS` now derived constants instead of hardcoded `0.75`/`11.25` literals), grew the window to 870×670 to compensate (nav/content_col end up larger), and tightened `panel_vbox`'s margin from 2px to 1px. Quantified all of this with real pixel scans rather than eyeballing: confirmed geometry grew exactly as expected (content_col 686px wide = 866 − 180 nav), and confirmed the border genuinely halved its rendered footprint (old: one full-colour pixel + one partial-blend pixel at a straight edge; new: a single partial-blend pixel, verified on the live `OptionsWindow` class itself, not just an isolated repro) — the user's report that it "doesn't look thinner" is most likely the moderate colour contrast between `SYS_STROKE_DIVIDER` and `SYS_BG_SECONDARY` making a real, measured pixel-level change hard to perceive at normal viewing distance, not a sign the change didn't take. **870×670 has not yet been re-confirmed seam-free on a real screen** — flagged to the user as outstanding, since it's a different value than the 850×650 already confirmed clean and the seam is proven to be size-sensitive.
